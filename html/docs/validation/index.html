These processors can only be used in the validation section of the resource.

They are specifically to restrict access to your resource. You can restrict access by valid token (anyone who is registered, role (anyone who has been given a specific role for your application) & user/role (specific users with roles for your application).

Token

Validate the request, requiring the user to have a valid token.

  • Inputs
    • Token
      The consumers token. Can be a processor.

Example (take the get var token and validate that in the security processor Token):

security:
  processor: token
  meta:
    id: 1
    token:
      processor: varPost
      meta:
        id: 2
        name: token

TokenUser

Validate the request, requiring the user to have a valid token and is in an array of usernames.

  • Inputs
    • Token
      The consumers token. Can be a processor.
    • Usernames
      one or more usernames. Can be processors or literals

Example (take the get var token and validate that in the security processor TokenUser):

security:
  processor: tokenUser
  meta:
    id: 1
    token:
      processor: varPost
      meta:
        id: 2
        name: token

TokenRole

Validate the request, requiring the user to have a valid token and is having a defined role.

  • Inputs
    • Token
      The consumers token. Can be a processor.
    • Role
      The consumer’s role. Can be processors or literals

Example (take the get var token and validate that in the security processor TokenRole, and only allowing access if that user has developer of consumer role):

security:
  processor: tokenConsumer
  meta:
    id: 1
    token:
      processor: varPost
      meta:
        id: 2
        name: token
    role: 
      - developer
      - consumer
